1. Overview

Write Dockerfiles that package apps predictably. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Python API Dockerfile

A minimal FastAPI-style Dockerfile.

FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

Build and run

Build tags make images easier to reference.

docker build -t learning-api:dev .
docker run --rm -p 8000:8000 learning-api:dev

5. Practical Workflow

  1. Choose a small official base image.
  2. Set WORKDIR.
  3. Copy dependency files first for caching.
  4. Install dependencies.
  5. Copy source code.
  6. Define CMD for runtime.
  7. Create .dockerignore.

6. Common Mistakes

Avoid these traps
  • Copying the entire project before installing dependencies and ruining cache.
  • Including .env or secrets in the image.
  • Using latest tags blindly.
  • Running as root in sensitive production contexts.
  • Confusing RUN and CMD.
