Testing & Debugging: pytest, Mocking & Profiling

Build confidence with tests, isolate bugs with evidence, and measure performance before optimizing.

1. Mental Model: Tests Are Executable Confidence

Testing is not only about proving code is correct. It is about making future changes safer. A good test suite lets you refactor without sweating like the production server is watching you. Debugging is the skill of finding the mismatch between what you thought the code did and what it actually did.

2. pytest Basics

Pytest discovers files named test_*.py or *_test.py, then runs functions named test_*. Use simple assert statements. Keep tests small and specific.

# file: calculator.py
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("b cannot be zero")
    return a / b
# file: test_calculator.py
import pytest
from calculator import add, divide

def test_add_two_numbers():
    assert add(2, 3) == 5

def test_divide_by_zero_raises_error():
    with pytest.raises(ValueError):
        divide(10, 0)

3. Arrange, Act, Assert

A readable test usually has three parts: arrange the input, act by calling the function, assert the expected behavior. This structure prevents test spaghetti.

def normalize_name(name):
    return " ".join(name.strip().title().split())

def test_normalize_name_removes_extra_spaces():
    # Arrange
    raw_name = "  galih   wijaya  "

    # Act
    result = normalize_name(raw_name)

    # Assert
    assert result == "Galih Wijaya"

4. Parametrized Tests

When the same logic should pass for many cases, use @pytest.mark.parametrize. It keeps tests compact without hiding the cases.

import pytest

@pytest.mark.parametrize(
    "score, expected",
    [
        (90, "A"),
        (80, "B"),
        (70, "C"),
    ],
)
def test_grade(score, expected):
    assert grade(score) == expected

5. Fixtures: Reusable Setup

Fixtures provide reusable setup for tests. They are great for temporary files, sample data, test clients, and initialized objects.

import pytest

@pytest.fixture
def sample_user():
    return {"name": "Alya", "role": "admin"}

def test_user_has_role(sample_user):
    assert sample_user["role"] == "admin"

tmp_path fixture

def test_write_report(tmp_path):
    report_path = tmp_path / "report.txt"
    report_path.write_text("hello", encoding="utf-8")

    assert report_path.read_text(encoding="utf-8") == "hello"

6. Mocking External Dependencies

Mocking replaces slow, flaky, or external dependencies. Use it for APIs, databases, emails, payment gateways, and random/time behavior. Do not mock the thing you are trying to test. That is like grading your own exam with vibes.

from unittest.mock import Mock

def notify_user(user_id, sender):
    sender.send(user_id, "Welcome")
    return True

def test_notify_user_calls_sender():
    sender = Mock()

    result = notify_user(123, sender)

    assert result is True
    sender.send.assert_called_once_with(123, "Welcome")

7. Debugging Techniques

  • Read the traceback from bottom to top to find the exact failing line.
  • Print or log the type and value of suspicious variables.
  • Reproduce the bug with the smallest input possible.
  • Use breakpoint() to inspect state interactively.
  • Write a failing test before fixing the bug, then make it pass.
def calculate_discount(price, rate):
    breakpoint()  # opens pdb when this line runs
    return price * rate

calculate_discount(100, 0.2)

8. Logging During Debugging

Use logging when you need persistent, leveled, contextual messages. Use logger.exception() inside exception handlers to keep the traceback.

import logging

logger = logging.getLogger(__name__)

def parse_int(text):
    try:
        return int(text)
    except ValueError:
        logger.exception("Failed to parse integer: %r", text)
        raise

9. Profiling: Find Slow Code With Evidence

Do not optimize based on guesswork. First measure. Use timeit for small snippets and cProfile for whole programs.

import timeit

elapsed = timeit.timeit(
    stmt="sum(range(1000))",
    number=10_000,
)
print(elapsed)
# Run from terminal:
# python -m cProfile -s cumulative your_script.py

10. What to Test

  • Normal cases: expected everyday input.
  • Edge cases: empty input, one item, large input, boundary values.
  • Error cases: invalid input, missing files, failed API response.
  • Contracts: output type, invariants, important side effects.
  • Regression cases: bugs that happened once and must not return.

11. Common Mistakes

  • Testing implementation details instead of behavior.
  • Writing one giant test that fails for ten possible reasons.
  • Mocking too much until the test no longer resembles reality.
  • Ignoring flaky tests instead of fixing the source of nondeterminism.
  • Optimizing before profiling.
