Fine-tuning LLMs: LoRA, QLoRA & Evaluation

Adapt models carefully and measure whether you actually improved anything.

Supports LLM Advanced
OverviewConceptsCodeWorkflowMistakesQuick refPractice

1. Overview

Fine-tuning adapts a base model to a task, style, or domain. LoRA and QLoRA reduce training cost by updating small adapter weights instead of all model parameters. Evaluation checks whether the fine-tuned model is better, safer, and worth the complexity.

Mental model

Fine-tuning changes model behavior. RAG changes model context. Prompting changes instructions. Choose fine-tuning when the model needs to learn a stable behavior pattern, not when it just needs new facts.

What you should be able to do after this page

Understand when to fine-tune, dataset formatting, LoRA adapters, QLoRA memory savings, training configs, evaluation sets, RAGAS-style RAG metrics, and safety regression testing.

3. Hands-on Code

Instruction dataset shape

Fine-tuning data should match the inference format.

examples = [
    {
        "messages": [
            {"role": "system", "content": "You are a concise code tutor."},
            {"role": "user", "content": "Explain Python lists."},
            {"role": "assistant", "content": "A list is an ordered, mutable collection..."},
        ]
    }
]

LoRA config concept

Exact APIs vary, but the idea is choosing target modules and adapter rank.

lora_config = {
    "r": 16,
    "alpha": 32,
    "dropout": 0.05,
    "target_modules": ["q_proj", "v_proj"],
    "task_type": "CAUSAL_LM",
}
print(lora_config)

Evaluation harness

Keep a small stable set of prompts with expected qualities.

eval_prompts = [
    {"prompt": "Explain recursion to a beginner", "must_include": ["base case", "recursive case"]},
    {"prompt": "Return JSON with keys score and feedback", "must_be_json": True},
]

for item in eval_prompts:
    output = model_generate(item["prompt"])
    print(item["prompt"], output[:120])

4. Practical Workflow

Fine-tune only after a baseline prompt/RAG setup fails in a repeatable way. Otherwise you are adding training complexity for no reason.

  1. Define why prompting or RAG is insufficient.
  2. Collect high-quality examples matching production format.
  3. Split train/validation/test.
  4. Train adapters with conservative settings.
  5. Evaluate on held-out and regression prompts.
  6. Compare against base model and prompt baseline.
  7. Monitor safety, style drift, and factuality.

5. Common Mistakes

Traps to avoid
  • Fine-tuning to add changing facts that belong in RAG.
  • Training on messy or inconsistent examples.
  • Evaluating only on training examples.
  • Ignoring base-model regression failures.
  • Assuming lower training loss means better user experience.
