1. Overview

Track changes locally with init, add, commit, log, and diff. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Start tracking

Initialize or clone before working.

git init
git status
git add README.md
git commit -m "Initial commit

Inspect changes

Always inspect before committing. Future you deserves mercy.

git status
git diff
git diff --staged

Read history

Logs explain how the project changed.

git log --oneline --graph --decorate

5. Practical Workflow

  1. Check status often.
  2. Make small focused changes.
  3. Review diff before staging.
  4. Stage related files.
  5. Write meaningful commit messages.
  6. Use log to understand history.

6. Common Mistakes

Avoid these traps
  • Committing huge unrelated changes.
  • Not reading diff before commit.
  • Writing messages like update/fix/stuff.
  • Committing generated files or secrets.
  • Panicking before checking status.
