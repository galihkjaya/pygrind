1. Overview

Combine tables correctly using keys and relationship types. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

INNER JOIN

Use when only matched rows answer the question.

SELECT o.id, u.email, o.amount
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status = 'paid';

LEFT JOIN

Use when the left table rows must survive even when right-side data is missing.

SELECT u.id, u.email, p.plan_name
FROM users u
LEFT JOIN subscriptions s ON s.user_id = u.id AND s.active = true
LEFT JOIN plans p ON p.id = s.plan_id;

Many-to-many

Bridge tables store relationships between two entities.

SELECT s.name, c.title
FROM students s
JOIN enrollments e ON e.student_id = s.id
JOIN courses c ON c.id = e.course_id;

5. Practical Workflow

  1. Draw the relationship before coding.
  2. Pick the base table: which rows must survive?
  3. Choose INNER or LEFT based on that survival rule.
  4. Join on keys, not names.
  5. Check row counts after each join.
  6. Investigate duplicates instead of blaming Postgres.

6. Common Mistakes

Avoid these traps
  • Joining on text names instead of IDs.
  • Using INNER JOIN when unmatched rows matter.
  • Breaking LEFT JOIN with a WHERE filter on the right table.
  • Ignoring duplicate rows from many-to-many joins.
  • Not checking key uniqueness before merging tables.
