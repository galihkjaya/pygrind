Type Hints & Dataclasses

Write self-documenting Python with safer function contracts and clean data containers.

1. Mental Model: Hints for Humans and Tools

Type hints do not normally change how Python runs. They make code easier to read, easier to review, easier to autocomplete, and easier to check with tools like mypy or pyright. Think of them as documentation that your editor can understand.

def greet(name: str) -> str:
    return f"Hello, {name}!"

message = greet("Galih")
print(message)
Runtime vs static checking

Python will still run many type-wrong programs unless you use a checker. Type hints are a contract for developers and tools, not a magical force field.

3. Useful Types from typing

The typing module gives more expressive types. Use the most general type that your function truly needs. If you only loop over values, accept Iterable. If you need indexing, accept Sequence. If you need to mutate, accept list.

from collections.abc import Callable, Iterable, Sequence

def apply_all(values: Iterable[int], fn: Callable[[int], int]) -> list[int]:
    return [fn(value) for value in values]

def first(values: Sequence[str]) -> str:
    return values[0]

print(apply_all([1, 2, 3], lambda x: x * 2))

4. Type Aliases and Literal Values

Type aliases make complex types readable. Literal restricts a value to specific strings or numbers, useful for modes and configuration.

from typing import Literal, TypeAlias

UserId: TypeAlias = int
Role: TypeAlias = Literal["student", "mentor", "admin"]

UserRecord: TypeAlias = dict[str, str | int]

def set_role(user_id: UserId, role: Role) -> None:
    print(f"Setting user {user_id} to {role}")

set_role(10, "mentor")

5. Dataclasses: Classes for Data Records

A dataclass removes repetitive boilerplate from classes that mainly store data. It automatically creates __init__, __repr__, and equality methods by default.

from dataclasses import dataclass

@dataclass
class Student:
    name: str
    score: float
    passed: bool = False

student = Student(name="Alya", score=88.5)
print(student)
print(student.name)

6. field(), default_factory, frozen, slots

Use field(default_factory=...) for mutable defaults. Use frozen=True for immutable records. Use slots=True to reduce memory and prevent accidental new attributes.

from dataclasses import dataclass, field

@dataclass(slots=True)
class Course:
    title: str
    lessons: list[str] = field(default_factory=list)

    def add_lesson(self, lesson: str) -> None:
        self.lessons.append(lesson)

course = Course("Intermediate Python")
course.add_lesson("Type Hints")
print(course.lessons)
from dataclasses import dataclass

@dataclass(frozen=True)
class Point:
    x: float
    y: float

p = Point(1.0, 2.0)
# p.x = 10.0  # would raise an error because frozen=True

7. __post_init__ and Derived Values

__post_init__ runs after the dataclass-generated __init__. Use it for validation and derived fields.

from dataclasses import dataclass, field

@dataclass
class ExamResult:
    student: str
    score: float
    grade: str = field(init=False)

    def __post_init__(self):
        if not 0 <= self.score <= 100:
            raise ValueError("score must be between 0 and 100")
        self.grade = "pass" if self.score >= 75 else "fail"

result = ExamResult("Bima", 82)
print(result.grade)

8. Practical Patterns

  • Use dataclasses for config objects, parsed records, DTOs, and feature rows.
  • Use type hints on public functions first. Internal tiny helpers can be typed later.
  • Use str | None only when None is truly allowed.
  • Avoid Any unless you have a clear reason.
  • Use field(default_factory=list) for lists and dictionaries.
Classic bug

Never write lessons: list[str] = [] inside a dataclass. Mutable defaults should use field(default_factory=list).

9. Common Mistakes

  • Assuming type hints enforce runtime validation automatically.
  • Using list when Sequence or Iterable would make the function more flexible.
  • Over-typing throwaway scripts while leaving important APIs unclear.
  • Using dataclasses for behavior-heavy domain objects that need custom methods and invariants.
  • Forgetting to validate external data. JSON from users is not trustworthy just because your dataclass is cute.
