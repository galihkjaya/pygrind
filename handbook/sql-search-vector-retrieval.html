1. Overview

Build full-text search and pg_vector retrieval for RAG-style apps. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

4. Hands-on Code

Full-text search

Keyword search with ranking.

SELECT id, title,
       ts_rank(search_vector, plainto_tsquery('english', 'python testing')) AS rank
FROM documents
WHERE search_vector @@ plainto_tsquery('english', 'python testing')
ORDER BY rank DESC
LIMIT 10;

Vector table

Store document chunks and embeddings.

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE document_chunks (
  id bigserial PRIMARY KEY,
  document_id bigint NOT NULL,
  content text NOT NULL,
  embedding vector(1536)
);

Semantic retrieval

The app passes the query embedding as a parameter.

SELECT id, content
FROM document_chunks
ORDER BY embedding <=> $1
LIMIT 5;

5. Practical Workflow

  1. Use full-text search for lexical search.
  2. Chunk long documents before embedding.
  3. Store metadata with chunks.
  4. Generate query embeddings in app code.
  5. Retrieve top-k chunks by vector distance.
  6. Pass context to the LLM with source information.

6. Common Mistakes

Avoid these traps
  • Using vector search when full-text search is enough.
  • Embedding whole books as one vector.
  • Forgetting metadata needed for citations.
  • Mixing embeddings from different models.
  • Assuming top-1 retrieval is always enough.
