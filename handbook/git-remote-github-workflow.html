1. Overview

Push, pull, fetch, and connect local repositories to GitHub. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Clone and work

Cloning sets origin automatically.

git clone https://github.com/user/repo.git
cd repo
git status

Add remote

For a local repo, connect it to GitHub.

git remote add origin https://github.com/user/repo.git
git branch -M main
git push -u origin main

Fetch before deciding

Fetch lets you inspect remote updates before merging.

git fetch origin
git log --oneline --graph --decorate --all

5. Practical Workflow

  1. Clone or add remote.
  2. Check remote with git remote -v.
  3. Push branch with upstream.
  4. Fetch before risky pulls.
  5. Pull latest before starting new work.
  6. Resolve divergence intentionally.

6. Common Mistakes

Avoid these traps
  • Pulling blindly with uncommitted changes.
  • Pushing to wrong remote.
  • Not setting upstream.
  • Confusing fetch and pull.
  • Force pushing shared branches without agreement.
