Packaging & Project Structure

Turn scripts into installable, testable, maintainable Python projects.

1. Mental Model: From Script to Project

A script is one file that solves one immediate task. A project is code that can grow, be tested, reused, installed, and understood by other people. Packaging and structure are how you stop a codebase from becoming a folder named final_final_REAL_v3.py.

ConceptPurpose
PackageA directory of importable Python modules
ModuleA single .py file
pyproject.tomlProject metadata, dependencies, and build config
Virtual environmentIsolated Python dependencies
src layoutKeeps import behavior clean during development

2. Recommended src Layout

The src layout prevents accidental imports from the project root. It forces you to install the package in editable mode during development, which better matches real usage.

my_project/
├── pyproject.toml
├── README.md
├── .gitignore
├── src/
│   └── my_project/
│       ├── __init__.py
│       ├── core.py
│       ├── config.py
│       └── cli.py
├── tests/
│   ├── test_core.py
│   └── test_config.py
└── scripts/
    └── demo.py

3. pyproject.toml Basics

pyproject.toml is the modern standard place for Python project metadata and tool configuration. Different build tools exist, but the basic idea is the same: declare the project name, version, Python requirement, and dependencies.

[project]
name = "my-project"
version = "0.1.0"
description = "A small Python learning project"
readme = "README.md"
requires-python = ">=3.11"
dependencies = [
    "requests>=2.32",
]

[project.optional-dependencies]
dev = [
    "pytest>=8.0",
    "ruff>=0.5",
    "mypy>=1.10",
]

[project.scripts]
my-project = "my_project.cli:main"

[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.build_meta"

4. Virtual Environments and Editable Installs

A virtual environment isolates dependencies for one project. An editable install lets changes in src/ immediately affect the installed package without reinstalling every time.

# Create and activate environment
python -m venv .venv

# Windows PowerShell
.venv\Scripts\Activate.ps1

# macOS/Linux
source .venv/bin/activate

# Install project in editable mode with dev dependencies
python -m pip install -e ".[dev]"

5. Imports That Do Not Fight You

Inside a package, prefer absolute imports from the package root. Relative imports are okay for nearby modules, but excessive relative dots make code harder to move.

# src/my_project/core.py
from my_project.config import Settings

def run(settings: Settings) -> str:
    return f"Running in {settings.env} mode"
# src/my_project/config.py
from dataclasses import dataclass

@dataclass
class Settings:
    env: str = "dev"
    debug: bool = True

6. Command-Line Entry Points

A CLI entry point lets users run your project as a command instead of calling Python files by path. The [project.scripts] section maps a command name to a function.

# src/my_project/cli.py
from my_project.config import Settings
from my_project.core import run

def main() -> None:
    settings = Settings()
    print(run(settings))

if __name__ == "__main__":
    main()

7. Configuration and Secrets

Configuration should be separate from code. Secrets should not be committed. Use environment variables for API keys and deployment-specific values.

import os
from dataclasses import dataclass

@dataclass
class Settings:
    api_key: str
    debug: bool = False

    @classmethod
    def from_env(cls):
        api_key = os.environ.get("API_KEY")
        if not api_key:
            raise RuntimeError("API_KEY is required")
        return cls(api_key=api_key, debug=os.environ.get("DEBUG") == "1")
Do not commit secrets

Never commit API keys, tokens, service account JSON files, or passwords. Add local env files to .gitignore. Future-you and GitHub secret scanners will thank you.

8. Tooling: Format, Lint, Type Check, Test

A serious Python project usually has four quality gates: formatter, linter, type checker, and tests. You do not need all of them on day one, but knowing the roles helps you scale.

Tool typeExamplesCatches
Formatterblack, ruff formatStyle inconsistency
LinterruffUnused imports, common bugs

# Common commands
python -m pytest
python -m ruff check src tests
python -m ruff format src tests
python -m mypy src

9. Building and Distributing

For internal projects, editable install is often enough. For distribution, you build a wheel and source distribution. Publish only when the package name, versioning, license, README, and dependencies are ready.

python -m pip install build
python -m build

# Outputs usually appear in dist/
# dist/my_project-0.1.0-py3-none-any.whl
# dist/my_project-0.1.0.tar.gz

10. README That Helps Users

  • What the project does in one paragraph.
  • Installation command.
  • One minimal usage example.
  • Configuration needed, especially environment variables.
  • How to run tests.
  • Known limitations or project status.

11. Common Mistakes

  • Running files by path forever instead of making a package.
  • Putting tests inside random folders with unclear imports.
  • Committing virtual environments, caches, or secrets.
  • Pinning dependencies too strictly for libraries or too loosely for apps.
  • Skipping README examples, then wondering why nobody can use the project.
