1. Overview

Containerization packages code, dependencies, and runtime configuration so apps run consistently across machines. Deployment platforms then run the container or app in a public or private environment.

Mental model

A deployment artifact should answer: what code runs, what dependencies exist, what command starts it, what environment variables are required, and what resources are needed.

What you should be able to do after this page

Write Dockerfiles, manage environment variables, prepare requirements, deploy FastAPI/Gradio-style apps, and understand CPU/GPU/resource trade-offs.

3. Hands-on Code

Minimal Dockerfile for FastAPI

Keep images small and explicit.

FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .
EXPOSE 8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

Local Docker commands

Build and run locally before deploying.

# build image
docker build -t ai-api:latest .

# run container
docker run --rm -p 8000:8000 --env-file .env ai-api:latest

4. Practical Workflow

Deployment is easier when your app has a clear start command and zero hidden notebook state.

  1. Move app code into normal Python files.
  2. Create requirements.txt or pyproject.toml.
  3. Read secrets from environment variables.
  4. Write a Dockerfile or platform config.
  5. Run locally in a clean environment.
  6. Add health check endpoint.
  7. Deploy and monitor logs, latency, and errors.

5. Common Mistakes

Traps to avoid
  • Depending on local files not copied into the image.
  • Hardcoding API keys.
  • Huge images with unnecessary dependencies.
  • No health check.
  • Not testing the production start command locally.
