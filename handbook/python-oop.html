1. Mental Model: Objects Bundle State and Behavior

Object-oriented programming is a way to model a program as cooperating objects. An object usually has state, meaning data it remembers, and behavior, meaning methods it can perform. A class is the blueprint. An instance is one actual object created from that blueprint.

TermMeaningExample
ClassBlueprint for objectsclass User:
InstanceOne real objectuser = User()
AttributeData attached to an objectuser.name
MethodFunction attached to an objectuser.login()
Dunder methodSpecial method Python calls automatically__init__, __repr__
When OOP is useful

Use OOP when data and behavior naturally belong together. A BankAccount should know its balance and how to deposit. A Tokenizer should know its vocabulary and how to encode text.

class User:
    def __init__(self, username, email):
        self.username = username
        self.email = email
        self.is_active = True

    def deactivate(self):
        self.is_active = False

    def describe(self):
        return f"{self.username} <{self.email}>"

user = User("galih", "galih@example.com")
print(user.describe())
user.deactivate()
print(user.is_active)

2. Classes, __init__, and self

__init__ runs when a new object is created. self refers to the current instance. Without self, your method only has temporary local variables and the object cannot remember anything after the method ends.

class CourseProgress:
    def __init__(self, course_name):
        self.course_name = course_name
        self.completed_lessons = []

    def mark_complete(self, lesson):
        self.completed_lessons.append(lesson)

    def completion_count(self):
        return len(self.completed_lessons)

progress = CourseProgress("Intermediate Python")
progress.mark_complete("OOP")
progress.mark_complete("Generators")
print(progress.completion_count())

Instance attributes vs class attributes

Instance attributes belong to one object. Class attributes are shared by the class. Use class attributes for constants or shared configuration, not per-user mutable state.

class Student:
    school = "Python Academy"     # class attribute

    def __init__(self, name):
        self.name = name           # instance attribute

s1 = Student("Alya")
s2 = Student("Bima")
print(s1.school, s2.school)
print(s1.name, s2.name)
Mutable class attribute trap

Do not put mutable per-instance data like [] or {} as a class attribute. Every object will share the same list. That bug is sneaky, like group project chaos in object form.

# Bad: every cart shares one list
class BadCart:
    items = []

    def add(self, item):
        self.items.append(item)

# Good: every cart gets its own list
class Cart:
    def __init__(self):
        self.items = []

    def add(self, item):
        self.items.append(item)

3. Encapsulation and Properties

Encapsulation means controlling how state is accessed and changed. Python uses conventions instead of strict private fields. A leading underscore, like _celsius, means internal use. A @property lets callers use attribute syntax while your class still validates or computes values.

class Temperature:
    def __init__(self, celsius):
        self.celsius = celsius

    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("Temperature cannot be below absolute zero")
        self._celsius = value

    @property
    def fahrenheit(self):
        return self._celsius * 9 / 5 + 32

temp = Temperature(25)
print(temp.fahrenheit)
temp.celsius = 30
print(temp.fahrenheit)

4. Inheritance, super(), and Composition

Inheritance lets one class reuse and specialize another class. Use it when the relationship is truly “is a”. A PaidUser is a User. For “has a” relationships, prefer composition. A Course has lessons. A Trainer has a logger.

class User:
    def __init__(self, username):
        self.username = username

    def can_access(self, feature):
        return feature == "free"

class PaidUser(User):
    def __init__(self, username, plan):
        super().__init__(username)
        self.plan = plan

    def can_access(self, feature):
        return feature in {"free", "premium"}

user = PaidUser("galih", "pro")
print(user.username)
print(user.can_access("premium"))
# Composition example
class EmailSender:
    def send(self, to, subject):
        print(f"Sending {subject} to {to}")

class NotificationService:
    def __init__(self, sender):
        self.sender = sender

    def welcome(self, email):
        self.sender.send(email, "Welcome!")

service = NotificationService(EmailSender())
service.welcome("student@example.com")

5. Dunder Methods: Make Objects Feel Native

Dunder methods are special methods surrounded by double underscores. Python calls them automatically for built-in operations. You do not call obj.__len__() directly in normal code. You call len(obj), and Python routes it to __len__.

DunderCalled byPurpose
__repr__repr(obj)Developer-friendly representation
__str__str(obj)User-friendly representation
__len__len(obj)Object length
__eq__obj1 == obj2Equality behavior
__getitem__obj[key]Indexing or lookup
class Playlist:
    def __init__(self, name, songs):
        self.name = name
        self.songs = list(songs)

    def __len__(self):
        return len(self.songs)

    def __getitem__(self, index):
        return self.songs[index]

    def __repr__(self):
        return f"Playlist(name={self.name!r}, songs={len(self)} songs)"

playlist = Playlist("Focus", ["lofi", "ambient", "jazz"])
print(len(playlist))
print(playlist[0])
print(playlist)

6. OOP Decorators: @classmethod, @staticmethod, @property

Decorators change how methods behave. In classes, the three you will see often are @property, @classmethod, and @staticmethod. They look similar but have different jobs.

DecoratorFirst parameterUse it for
@propertyselfComputed attribute or validated attribute access
@classmethodclsAlternative constructors and class-level factories
@staticmethodNone automaticallyHelper function logically grouped inside class
class User:
    def __init__(self, username, email):
        self.username = username
        self.email = email

    @classmethod
    def from_email(cls, email):
        username = email.split("@")[0]
        return cls(username=username, email=email)

    @staticmethod
    def is_valid_email(email):
        return "@" in email and "." in email

user = User.from_email("galih@example.com")
print(user.username)
print(User.is_valid_email("bad-email"))

7. Designing Good Classes

  • A class should have one clear responsibility. If it does five unrelated jobs, split it.
  • Keep method names as verbs: save, predict, validate, transform.
  • Keep attribute names as nouns: name, items, model_path.
  • Use composition for flexible systems. Inject dependencies like loggers, clients, and model objects.
  • Do not over-OOP tiny scripts. Sometimes a function is cleaner. Not every variable needs a throne.
