1. Overview

Rewrite, clean, and recover history without nuking the repo. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Rebase feature branch

Rebase keeps feature history linear.

git switch feature/course-api
git fetch origin
git rebase origin/main

Safe undo with revert

Revert is safer for shared history because it adds an undo commit.

git revert abc123

Restore local file

Restore discards uncommitted changes in a file.

git restore app/main.py
git restore --staged app/main.py

5. Practical Workflow

  1. Use merge for simple shared collaboration.
  2. Use rebase to update a private feature branch.
  3. Never rebase public/shared commits casually.
  4. Use revert to undo committed shared changes.
  5. Use restore for local file changes.
  6. Use reflog when you think you lost commits.

6. Common Mistakes

Avoid these traps
  • Rebasing shared branches and confusing everyone.
  • Using reset --hard without checking status.
  • Using revert and expecting history to disappear.
  • Not knowing reflog exists.
  • Force pushing without coordination.
