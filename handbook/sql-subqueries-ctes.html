1. Overview

Use nested queries and WITH blocks to make complex SQL readable. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Scalar subquery

Compare each row to a calculated summary.

SELECT id, title, price
FROM courses
WHERE price > (SELECT AVG(price) FROM courses);

CTE pipeline

Name intermediate results so the query reads like a story.

WITH paid_orders AS (
  SELECT * FROM orders WHERE status = 'paid'
), revenue AS (
  SELECT customer_id, SUM(amount) AS total_revenue
  FROM paid_orders
  GROUP BY customer_id
)
SELECT *
FROM revenue
WHERE total_revenue > 1000000;

EXISTS

Use EXISTS when you only care that a related row exists.

SELECT u.id, u.email
FROM users u
WHERE EXISTS (
  SELECT 1
  FROM enrollments e
  WHERE e.user_id = u.id
);

5. Practical Workflow

  1. Write the final question first.
  2. Identify intermediate datasets.
  3. Use CTE names that describe meaning.
  4. Keep each CTE focused on one transformation.
  5. Run each stage while debugging.
  6. Optimize only after correctness is locked.

6. Common Mistakes

Avoid these traps
  • Nesting deeply until nobody can debug it.
  • Naming CTEs a, b, temp.
  • Assuming CTEs always improve performance.
  • Using IN when EXISTS communicates intent better.
  • Forgetting that subqueries can return multiple rows unexpectedly.
