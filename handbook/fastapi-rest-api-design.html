1. Overview

Design resources, status codes, pagination, and consistent responses. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Resource routes

Design around nouns and resources.

GET    /api/v1/courses
POST   /api/v1/courses
GET    /api/v1/courses/{course_id}
PATCH  /api/v1/courses/{course_id}
DELETE /api/v1/courses/{course_id}

Response shape

Choose a response convention and reuse it.

{
  "data": {"id": 1, "title": "SQL Basics"},
  "meta": {"request_id": "req_123"}
}

Pagination

List endpoints need paging before the database becomes chunky.

@app.get('/courses')
def list_courses(limit: int = 20, offset: int = 0):
    return {'data': [], 'pagination': {'limit': limit, 'offset': offset, 'total': 0}}

5. Practical Workflow

  1. Identify resources and relationships.
  2. Define only needed CRUD routes.
  3. Choose status codes deliberately.
  4. Standardize success and error shapes.
  5. Add pagination/filtering to list endpoints.
  6. Use versioned prefixes for public APIs.

6. Common Mistakes

Avoid these traps
  • Designing endpoints around UI buttons.
  • Returning 200 for everything.
  • No pagination until production melts.
  • Changing response fields without versioning.
  • Inconsistent naming like userId/user_id/idUser.
