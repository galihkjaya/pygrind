CI Basics with GitHub Actions

Run automated checks, tests, and Docker builds on every push or pull request.

1. Overview

Run automated checks, tests, and Docker builds on every push or pull request. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Python test workflow

Start CI with install and test.

name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install -r requirements.txt
      - run: pytest

Docker build check

Build image to catch broken Dockerfiles.

jobs:
  docker:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t app:test .

Use secrets

Secrets are injected at runtime, not committed.

- name: Run integration test
  env:
    API_KEY: ${{ secrets.API_KEY }}
  run: pytest tests/integration

5. Practical Workflow

  1. Create .github/workflows/ci.yml.
  2. Run on push and pull_request.
  3. Checkout code.
  4. Install language/runtime.
  5. Install dependencies.
  6. Run tests/lint/build.
  7. Use secrets for sensitive values.

6. Common Mistakes

Avoid these traps
  • No CI until final deployment day.
  • Putting real secrets in YAML.
  • Running slow integration tests on every tiny change without strategy.
  • Ignoring failing CI.
  • Not pinning major action versions.
