1. Overview

LLM APIs let applications send prompts and receive model outputs. Production apps usually need streaming for UX, structured outputs for reliability, and tool/function calling for connecting models to real systems.

Mental model

Treat an LLM provider as an interchangeable adapter. Your app should define one internal request/response shape, then map it to OpenAI, Groq, Gemini, or another provider.

What you should be able to do after this page

Understand API keys, chat messages, streaming, JSON/structured output, function calling, provider adapters, retries, rate limits, and client-side security trade-offs.

3. Hands-on Code

Provider-neutral interface

Define your app contract first, then write adapters.

from dataclasses import dataclass

@dataclass
class LLMRequest:
    system: str
    user: str
    json: bool = False

@dataclass
class LLMResponse:
    text: str
    provider: str
    model: str

def call_llm(request: LLMRequest, provider: str) -> LLMResponse:
    if provider == "groq":
        return call_groq(request)
    if provider == "gemini":
        return call_gemini(request)
    if provider == "openai":
        return call_openai(request)
    raise ValueError(f"Unknown provider: {provider}")

Streaming mental model

Streaming improves perceived latency by rendering partial output as it arrives.

def render_stream(chunks):
    buffer = []
    for chunk in chunks:
        token = chunk.get("text", "")
        buffer.append(token)
        print(token, end="", flush=True)
    return "".join(buffer)

Tool calling loop

The model proposes a tool call; your code executes it; then the result goes back to the model.

tools = {
    "get_weather": lambda city: {"city": city, "temp_c": 30},
}

model_response = {
    "tool_name": "get_weather",
    "arguments": {"city": "Semarang"},
}

name = model_response["tool_name"]
args = model_response["arguments"]
tool_result = tools[name](**args)
print(tool_result)

Structured output validation

Always validate JSON before trusting it.

import json
from pydantic import BaseModel, ValidationError

class Review(BaseModel):
    score: int
    feedback: str
    passed: bool

raw = '{"score": 82, "feedback": "Good structure", "passed": true}'
try:
    review = Review.model_validate(json.loads(raw))
    print(review)
except (json.JSONDecodeError, ValidationError) as e:
    print("Invalid model output", e)

4. Practical Workflow

Provider APIs change over time, so isolate provider-specific code and keep your app logic stable.

  1. Store API keys securely and avoid committing them.
  2. Create one internal LLMInput/LLMOutput shape.
  3. Write separate adapters per provider.
  4. Use streaming for long visible responses.
  5. Use structured output for UI rendering or grading.
  6. Validate all JSON and tool arguments.
  7. Add retries, timeouts, logging, and cost tracking.

5. Common Mistakes

Traps to avoid
  • Hardcoding provider-specific response parsing everywhere.
  • Putting secret API keys into public repos.
  • Trusting JSON without validation.
  • Letting the model execute tools directly instead of your code controlling execution.
  • Ignoring rate limits, retries, and cost.
