Modules & Packages

Organize Python code into reusable modules, scalable packages, clean imports, and dependency-managed projects.

1. Mental Model: Split Code into Reusable Pieces

A module is a Python file you can import. A package is a folder of modules. These let you organize code, reuse logic, avoid giant files, and share functionality across projects. Once your code grows beyond one file, modules stop being optional.

Simple definitions

  • Module: one .py file, such as utils.py.
  • Package: a folder containing modules, usually with __init__.py.
  • Import: telling Python to load code from another module.
  • Library: a collection of modules/packages built for reuse.

2. Import Basics

Python already ships with many standard library modules. You can import them without installing anything.

import math
import random
from datetime import datetime

print(math.sqrt(16))
print(random.randint(1, 10))
print(datetime.now())

Different import styles

# Import the whole module
import math
print(math.pi)

# Import specific names
from math import pi, sqrt
print(pi)
print(sqrt(25))

# Alias a module
import statistics as stats
print(stats.mean([10, 20, 30]))

# Alias a specific function
from pathlib import Path as P
print(P.cwd())
StyleGood forTradeoff
import moduleClarityNeed module prefix
from module import nameFrequently used functionsCan hide origin
import module as aliasCommon conventionsAlias must be obvious
from module import *Almost neverPollutes namespace
❌ Avoid wildcard imports from module import * makes it hard to know where names came from and can overwrite existing names. It looks convenient, then becomes debugging DLC.

3. Your Own Local Modules

If you create a file called calculator.py, you can import it from another file in the same folder.

# calculator.py

def add(a, b):
    return a + b


def multiply(a, b):
    return a * b
# main.py

import calculator

print(calculator.add(3, 4))
print(calculator.multiply(5, 6))

Import specific functions from your module

# main.py

from calculator import add, multiply

print(add(3, 4))
print(multiply(5, 6))
Keep modules focused A good module has one clear responsibility: validation functions, file helpers, feature engineering, math utilities, API clients, and so on.

4. The __name__ == "__main__" Pattern

When Python runs a file directly, its __name__ is "__main__". When the file is imported, its __name__ is the module name. This pattern prevents test/demo code from running during import.

# greetings.py

def greet(name):
    return f"Hello, {name}!"


if __name__ == "__main__":
    # This only runs when you execute: python greetings.py
    print(greet("Alya"))
# main.py

from greetings import greet

# The demo print inside greetings.py does not run here
print(greet("Bima"))

5. How Python Finds Modules

When you import something, Python searches through a list of locations. The first matching name wins.

import sys

for path in sys.path:
    print(path)

The important beginner idea: Python usually checks your current project folder first. This means bad file names can accidentally shadow standard library modules.

❌ Do not name your file like a standard library module Avoid names like random.py, json.py, csv.py, math.py, or datetime.py. If you do, Python may import your file instead of the real library.

6. Packages: Folders of Modules

A package is a folder that groups related modules. This is how you scale from scripts into maintainable projects.

project/
├── main.py
└── myapp/
    ├── __init__.py
    ├── validators.py
    ├── file_utils.py
    └── features.py
# myapp/validators.py

def is_valid_score(score):
    return 0 <= score <= 100
# myapp/features.py

from myapp.validators import is_valid_score


def normalize_score(score):
    if not is_valid_score(score):
        raise ValueError("Invalid score")
    return score / 100
# main.py

from myapp.features import normalize_score

print(normalize_score(85))

What is __init__.py?

__init__.py marks a folder as a regular Python package and can expose selected names for easier imports. It can be empty, especially at the beginning.

# myapp/__init__.py

from myapp.features import normalize_score

# main.py can now do:
# from myapp import normalize_score

7. Absolute vs Relative Imports

Absolute imports start from the project/package root. Relative imports use dots to refer to nearby modules inside the same package.

Absolute import

from myapp.validators import is_valid_score

Usually clearer and recommended for most beginner/intermediate projects.

Relative import

from .validators import is_valid_score

Only works inside packages. Useful for internal package modules.

⚠️ Relative imports can fail when running files directly If a file inside a package uses relative imports, run the package from the project root instead of running that internal file directly.

8. Third-Party Packages, pip, and Virtual Environments

The standard library is built in. Third-party packages must be installed. For real projects, install them inside a virtual environment so each project gets its own dependency space.

# Create virtual environment
python -m venv .venv

# Activate on Windows
.venv\Scripts\activate

# Activate on macOS/Linux
source .venv/bin/activate

# Install package
pip install requests

# Save dependencies
pip freeze > requirements.txt

# Reinstall dependencies later
pip install -r requirements.txt

Using installed packages

import requests

response = requests.get("https://example.com")
print(response.status_code)
Project hygiene Commit requirements.txt, but usually do not commit the .venv folder. The requirements file is the recipe; the virtual environment is the cooked food.

10. Circular Imports

A circular import happens when two modules import each other. This often means your code responsibilities are tangled.

# users.py
from emails import send_welcome_email


def create_user(email):
    send_welcome_email(email)

# emails.py
from users import create_user  # circular import risk

How to fix the design

  • Move shared logic into a third module, such as services.py or utils.py.
  • Keep low-level modules independent from high-level workflow modules.
  • Import inside a function only as a temporary workaround, not your default design.
  • Review whether each module has a clear responsibility.

11. Common Mistakes

❌ Running package modules from the wrong location

If imports fail, check your working directory and run from the project root. Many import bugs are secretly “you are standing in the wrong folder” bugs.

❌ Putting too much code at import time 
# Bad: this runs immediately when imported
print("Loading data...")
data = load_huge_file()

# Better: put work inside a function
def main():
    data = load_huge_file()

if __name__ == "__main__":
    main()
⚠️ Importing from deep internal paths too often If users of your package must import from five folders deep, expose stable functions in __init__.py or redesign the structure.
