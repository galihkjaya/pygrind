1. Overview

Use JSONB, arrays, UUIDs, timestamps, and enums without wrecking your schema. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

JSONB metadata

JSONB works well for flexible attributes.

CREATE TABLE events (
  id bigserial PRIMARY KEY,
  user_id bigint NOT NULL,
  event_type text NOT NULL,
  properties jsonb NOT NULL DEFAULT '{}',
  created_at timestamptz NOT NULL DEFAULT now()
);

SELECT * FROM events
WHERE properties ->> 'source' = 'mobile';

GIN index

GIN supports containment queries on JSONB.

CREATE INDEX idx_events_properties
ON events USING GIN (properties);

SELECT * FROM events
WHERE properties @> '{"plan":"pro"}';

Array tags

Arrays are okay for small simple lists.

CREATE TABLE articles (
  id bigserial PRIMARY KEY,
  title text NOT NULL,
  tags text[] NOT NULL DEFAULT '{}'
);

SELECT * FROM articles
WHERE 'python' = ANY(tags);

5. Practical Workflow

  1. Use normal columns for stable, query-heavy fields.
  2. Use JSONB for flexible metadata.
  3. Use arrays only for small same-type lists.
  4. Prefer timestamptz for real-world time.
  5. Add indexes based on real query patterns.
  6. Document JSON structure in app validation.

6. Common Mistakes

Avoid these traps
  • Using JSONB to avoid database design entirely.
  • Storing timestamps without timezone.
  • Using arrays instead of relationship tables when metadata is needed.
  • Indexing every JSON key blindly.
  • Mixing inconsistent JSON shapes in one column.
