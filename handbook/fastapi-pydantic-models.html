1. Overview

Validate request bodies and control response schemas. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

4. Hands-on Code

Request model

FastAPI validates incoming JSON automatically.

from pydantic import BaseModel, Field

class CourseCreate(BaseModel):
    title: str = Field(min_length=3, max_length=100)
    level: str
    estimated_hours: int = Field(gt=0, le=500)

@app.post('/courses', status_code=201)
def create_course(payload: CourseCreate):
    return {'received': payload.model_dump()}

Separate input and output

Different models prevent accidental data leaks.

class UserCreate(BaseModel):
    email: str
    password: str

class UserOut(BaseModel):
    id: int
    email: str

@app.post('/users', response_model=UserOut)
def create_user(payload: UserCreate):
    return {'id': 1, 'email': payload.email, 'password_hash': 'hidden'}

Nested response

Nested schemas document complex response shapes.

class LessonOut(BaseModel):
    id: int
    title: str

class CourseOut(BaseModel):
    id: int
    title: str
    lessons: list[LessonOut]

5. Practical Workflow

  1. Define request and response models before business logic.
  2. Separate create, update, and output schemas.
  3. Use Field constraints for validation.
  4. Use response_model to hide internal fields.
  5. Add examples for complex payloads.
  6. Let FastAPI generate docs from schemas.

6. Common Mistakes

Avoid these traps
  • Using one model for input/output/internal data.
  • Returning password hashes accidentally.
  • Accepting arbitrary dicts when shape is known.
  • Putting business logic inside schema classes.
  • Forgetting optional fields in PATCH models.
