1. Overview

Work safely with feature branches and resolve merge conflicts. This handbook focuses on practical understanding first: what the tool does, why it exists, how to write the syntax, and how to avoid mistakes that usually waste hours.

Read it like a field guide. You do not need to memorize every command on first pass. You need to understand the shape of the workflow and know which pattern to reach for when a real project asks for it.

2. Mental Model

How to think about this topic

The core mental model: every command or construct has an input, a transformation, and an output. Strong engineers do not just copy syntax. They predict what should happen, run the smallest possible check, then scale the pattern carefully.

Learning target

After this page, you should be able to explain the purpose of the topic, write the main syntax from scratch, debug common failure modes, and apply it inside a small backend or AI-engineering project.

5. Practical Workflow

  1. Create a feature branch per task.
  2. Commit focused changes.
  3. Update main before merging.
  4. Run tests before merge.
  5. When conflict appears, open file and choose final content.
  6. Stage resolved files and complete merge.

6. Common Mistakes

Avoid these traps
  • Working directly on main for risky tasks.
  • Deleting conflict markers without understanding content.
  • Merging without tests.
  • Keeping long-lived branches stale.
  • Resolving conflict by blindly accepting one side.
