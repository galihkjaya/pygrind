Agents & Agentic Workflows: LangChain, LlamaIndex & ReAct

Let models plan and use tools, but keep them on a leash. A polite leash.

1. Overview

Agentic workflows let an LLM choose actions such as searching, calling APIs, writing files, or asking tools. They are useful for multi-step tasks but require guardrails, state management, and observability.

Mental model

An agent loop usually follows: observe task, think/plan, choose tool, execute tool, observe result, continue or answer. The model decides the next action, but your application defines allowed tools and safety limits.

What you should be able to do after this page

Understand ReAct, tools, planners, memory, state, tool schemas, orchestration frameworks, failure modes, and when not to use agents.

3. Hands-on Code

Tool schema

Tools need clear names, arguments, and descriptions.

def search_handbook(query: str, top_k: int = 3) -> list[dict]:
    """Search handbook chunks and return top matching passages."""
    return retriever.search(query, top_k=top_k)

tools = {
    "search_handbook": search_handbook,
}

Simple agent loop

Keep loops bounded. Infinite agent loops are just expensive confusion.

max_steps = 5
state = {"messages": [], "task": "Find the best handbook page for async Python"}

for step in range(max_steps):
    action = model_choose_action(state, tools=list(tools))
    if action["type"] == "final":
        print(action["answer"])
        break
    if action["tool"] not in tools:
        raise ValueError("Tool not allowed")
    observation = tools[action["tool"]](**action["args"])
    state["messages"].append({"action": action, "observation": observation})

Guardrail pattern

Validate every tool call before execution.

def safe_execute(action):
    allowed = {"search_handbook", "read_file"}
    if action["tool"] not in allowed:
        return {"error": "tool_not_allowed"}
    if len(str(action.get("args", {}))) > 2000:
        return {"error": "args_too_large"}
    return tools[action["tool"]](**action["args"])

4. Practical Workflow

Use agents when the task truly requires dynamic multi-step decisions. For simple extraction or classification, a normal prompt is cheaper and more reliable.

  1. Define task and decide whether agentic behavior is necessary.
  2. Create small, reliable tools with typed inputs.
  3. Expose only safe tools to the agent.
  4. Set max steps, timeouts, and cost limits.
  5. Log every action and observation.
  6. Validate tool arguments.
  7. Evaluate task success and failure modes.

5. Common Mistakes

Traps to avoid
  • Using agents for tasks a single deterministic function could solve.
  • Giving tools too much power without validation.
  • No max step limit. Wallet says bye.
  • No logging, making failures impossible to debug.
  • Letting memory grow forever without summarization.
